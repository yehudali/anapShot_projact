## הבעיה שאנחנו פותרים

דמיינו את הסצנריו הבא:

**כוננת מד"א** מקבלת קריאה לאירוע רב-נפגעים בצומת הכניסה לעיר.
היא יודעת שיש עוד 6 כוננים פעילים באזור — אבל **היא לא יודעת איפה הם**.
האם הכוונן הקרוב נוסע כבר לשם? מה ה-ETA שלו? מאיזה כיוון הוא מגיע?
כל שניה חשובה — והיא מבזבזת אותן על שיחות טלפון.

**מפקד פלוגה בשדה הקרב** מקבל משימה דחופה. הוא צריך לדעת בזמן אמת:
- איפה כל הלוחמים שלו
- מי הכי קרוב לנקודת המטרה
- האם יש פריסה שמשאירה נקודה חשופה

**מנהל תחנת מוניות** רואה שלקוח ממתין כבר 20 דקות ברחוב גשום.
הוא יודע שיש לו 5 נהגים "זמינים" — אבל באיזה רחוב בדיוק? מי מהם הכי קרוב?

## הפתרון: מודעות מצבית בזמן אמת

**AnapShot** היא מערכת שנותנת תשובה אחת לכל הסצנריים האלה:

> **"תמיד תדע בדיוק איפה כל אחד מהאנשים שלך, בזמן אמת, ברגע שאירוע מתחיל."**

ברגע שנוצר "אירוע" במערכת — כל המכשירים הפעילים מתעוררים ומתחילים לשדר את מיקומם.
המידע מופיע על מפה חיה, מתעדכן כל שתי שניות, עד שהאירוע נסגר.

---

## מי משתמש במערכת

### תפקידים (Roles)

| תפקיד | מי זה | מה הוא עושה |
|---|---|---|
| **ADMIN** | מנהל/מפקד | פותח אירועים, מנהל מכשירים ומשתמשים, רואה הכל |
| **MANAGER** | קצין מבצעים / מוקדן | צופה בפריסה הכוללת, לא יוצר אירועים |
| **DEVICE** | הכוננ/לוחם/נהג עצמו | שולח מיקום GPS בלבד (לא רואה UI) |


## האדריכלות מנקודת מבט הפרונט

### מה הפרונט צריך לבנות

```
┌─────────────────────────────────────────────────────────────┐
│                      ממשק המשתמש                            │
│                                                             │
│  ┌─────────────┐  ┌─────────────────┐  ┌────────────────┐  │
│  │  לוח אירועים │  │   מפה חיה       │  │ ניהול מכשירים  │  │
│  │  (Events)   │  │ (Live Map)      │  │  (Devices)    │  │
│  └─────────────┘  └─────────────────┘  └────────────────┘  │
└─────────────────────────────────────────────────────────────┘
          │                  │                    │
          ▼                  ▼                    ▼
    REST API calls     WebSocket             REST API calls
    (פתיחה/סגירה)    (עדכוני מיקום)       (CRUD מכשירים)
```

### החיבור לבאקנד — שני ערוצים

#### 1. REST API — לפעולות רגילות
```
POST   /api/v1/auth/login          ← התחברות
POST   /api/v1/events              ← פתיחת אירוע חדש
PUT    /api/v1/events/{id}/close   ← סגירת אירוע
GET    /api/v1/devices             ← רשימת כל המכשירים
GET    /api/v1/events/{id}/history ← היסטוריית מסלול (אחרי אירוע)
```

#### 2. WebSocket — לעדכונים חיים
```javascript
// פתיחת חיבור WebSocket לאירוע
const ws = new WebSocket(`ws://server/ws/events/${eventId}?token=${jwtToken}`)

ws.onmessage = (msg) => {
  const data = JSON.parse(msg.data)
  // data.locations = מערך של כל המכשירים עם קואורדינטות עכשוויות
  // data.status = "active" | "closed"
  updateMapMarkers(data.locations) // עדכן סמנים על המפה
}
```

### פורמט הנתונים שמגיע מה-WebSocket
```json
{
  "status": "active",
  "locations": [
    {
      "device_id": "6507f1f77bcf86cd799439011",
      "latitude": 31.7683,
      "longitude": 35.2137,
      "timestamp": "2026-03-25T14:32:10",
      "accuracy": 7.5
    },
    {
      "device_id": "6507f1f77bcf86cd799439012",
      "latitude": 31.7695,
      "longitude": 35.2150,
      "timestamp": "2026-03-25T14:32:09",
      "accuracy": 12.0

```

כל `device_id` הוא סמן אחד על המפה. הנתונים מתחלפים כל **2 שניות**.

## מה קורה מאחורי הקלעים (ולמה זה חשוב לפרונט)

| מה קורה | איך זה משפיע על הפרונט |
|---|---|
| מכשיר שולח GPS כל 5 שניות | הפרונט יראה עדכון כל 2 שניות (קצב הWebSocket) |
| מכשיר לא שלח מעל 30 שניות | הסטטוס שלו יהיה `UNREACHABLE` — כדאי להציג בצבע שונה על המפה |
| אירוע נסגר | ה-WebSocket ישלח `"status": "closed"` ואז יסגר — הפרונט צריך לטפל בזה |
| מכשיר `INACTIVE` | לא מקבל wake-up, לא יופיע במפה |

## מסכי הפרונט המרכזיים

### 1. מסך התחברות
- Login form → שמירת JWT token
- ניתוב לפי Role: ADMIN ← לוח ניהול מלא, MANAGER ← מפה בלבד

### 2. לוח אירועים (Dashboard)
- רשימת אירועים פעילים וסגורים
- כפתור "פתח אירוע חדש" (ADMIN בלבד)
- כפתור "סגור אירוע" (ADMIN בלבד)
- קישור לכניסה למפה החיה של אירוע

### 3. מפה חיה (Live Map) — **הלב של המערכת**
- מפה אינטראקטיבית (Leaflet / Google Maps / Mapbox)
- חיבור WebSocket לאירוע הפעיל
- סמן לכל מכשיר עם tooltip: שם המכשיר, זמן דיווח אחרון
- **צבעים לפי סטטוס:**
  - 🟢 ירוק — דיווח תקין (פחות מ-15 שניות)
  - 🟡 צהוב — עיכוב (15-30 שניות)
  - 🔴 אדום — UNREACHABLE (מעל 30 שניות)
- אינדיקטור "מחובר / מנותק" לחיבור ה-WebSocket

### 4. ניהול מכשירים (ADMIN)
- רשימת מכשירים עם סטטוס (ACTIVE/INACTIVE/UNREACHABLE)
- יצירת מכשיר חדש — מקבלים בחזרה `api_key` (להראות פעם אחת!)
- הפעלה/השבתה של מכשיר

### 5. היסטוריית אירוע (אחרי סגירה)
- Replay של מסלול כל מכשיר על המפה
- ציר זמן אינטראקטיבי

## הנחות ייעוד שכדאי לדעת

- **לא mobile-first בלבד** — מנהלים יעבדו מ-desktop/tablet, אבל שוקלים גם מובייל
- **ביצועים קריטיים** — המפה חייבת להיות responsive גם עם 50+ מכשירים בו-זמנית
- **חיבור לא יציב** — חייבים לטפל ב-WebSocket reconnect (שטח הפעולה של מד"א/צבא)
- **UX חירום** — הממשק חייב להיות ברור ופשוט תחת לחץ. אין מקום ל-UX מורכב.

## Flow מהיר למפתח — מ-0 לאירוע פעיל

```
1. POST /api/v1/auth/login
   → שמירת token

2. POST /api/v1/devices (x5, לדוגמה)
   → יצירת מכשירים פעילים

3. POST /api/v1/events { "name": "אירוע ניסיון", "description": "..." }
   → המערכת מפעילה את כל המכשירים אוטומטית

4. WebSocket: ws://localhost:8000/ws/events/{event_id}?token=...
   → תתחיל לקבל locations כל 2 שניות!

5. PUT /api/v1/events/{id}/close
   → WebSocket נסגר, האירוע מסתיים
```

## למה הארכיטקטורה הזו?

| בעיה | פתרון |
|---|---|
| צריך לעדכן 50 מכשירים בו-זמנית | **Kafka** — message broker שמבטיח שכל מכשיר יקבל את האות |
| חיבורי WebSocket רבים צורכים זיכרון | **Redis** — מחזיק snapshot אחד בלבד לכל אירוע (לא את כל ההיסטוריה) |
| צריך לשמור היסטוריה מלאה | **MongoDB** — כל דיווח GPS נשמר לצמיתות |
| גמישות בחיבור/ניתוק מכשירים | **API Key** לכל מכשיר — עצמאי מה-JWT של המשתמשים |

## שאלות נפוצות מהפרונט

**ש: אני צריך לשלוח GPS מהפרונט?**
ת: לא. GPS מגיע רק ממכשירים פיזיים (Role: DEVICE) דרך `X-Api-Key`. הפרונט רק *צופה*.

**ש: איך אני יודע אם מכשיר "מת"?**
ת: בדוק את ה-`timestamp` בנתוני המיקום. אם עבר יותר מ-30 שניות — תציג כ-UNREACHABLE.

**ש: האם WebSocket יכול להתנתק?**
ת: כן. חייב לממש reconnect logic. הבאקנד לא שומר state של החיבור.

**ש: מה קורה כשאני פותח WebSocket לאירוע סגור?**
ת: הבאקנד ישלח `{ "status": "closed" }` מיד ויסגור את החיבור.

**ש: האם יכולים להיות כמה ADMIN/MANAGER מחוברים בו-זמנית?**
ת: כן. כל אחד מקבל WebSocket עצמאי, כולם רואים את אותם נתונים.
